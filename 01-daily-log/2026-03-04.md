========================================
Object, OCP, equals() 정리
========================================


[1] Object가 없다면 생기는 문제
----------------------------------------
만약 Object 같은 공통 부모가 없고,
또 Object가 제공하는 toString()도 없다면
서로 관계없는 객체들의 정보를 하나의 방식으로 출력하기 어려워진다.

예를 들어 Car, Dog 같은 클래스가 있을 때
각 객체마다 전용 출력 메서드를 따로 만들어야 한다.

즉,
- Car 출력용 메서드
- Dog 출력용 메서드
- 나중에 Person, Book, User가 추가되면 또 전용 메서드 추가

이렇게 되면 객체 종류가 늘어날수록
출력 코드도 계속 늘어난다.


[2] 구체적인 것에 의존하는 구조의 문제
----------------------------------------
공통 부모 없이 각 타입별로 따로 처리하면
출력 클래스는 Car, Dog 같은 "구체적인 클래스"에 직접 의존하게 된다.

이 구조의 문제점
- 새로운 클래스가 추가될 때마다 메서드를 계속 추가해야 한다.
- 기존 코드 수정이 반복된다.
- 유지보수가 어려워진다.
- 확장성이 떨어진다.

즉, 클라이언트 코드가
구체적인 타입에 강하게 묶이게 된다.


[3] Object를 사용하면 해결되는 이유
----------------------------------------
자바는 모든 객체의 공통 부모인 Object를 제공한다.

그래서 출력 기능을 만들 때
구체적인 Car, Dog에 의존하지 않고
Object 하나만 사용해서 모든 객체를 받을 수 있다.

핵심은 2가지이다.

1. 다형적 참조
   - Object 타입으로 모든 객체를 받을 수 있다.

2. 메서드 오버라이딩
   - 각 클래스가 toString()을 오버라이딩하면
     자신의 정보를 각자 알맞게 표현할 수 있다.

즉,
출력하는 쪽은 Object만 알면 되고,
실제 출력 내용은 각 객체가 자기 책임으로 제공한다.


[4] 추상적인 것에 의존한다는 의미
----------------------------------------
ObjectPrinter 같은 구조는
Car, Dog 같은 구체적인 클래스에 의존하지 않고
상위 개념인 Object에 의존한다.

여기서 "추상적"이라는 말은
반드시 추상 클래스나 인터페이스만 의미하는 것은 아니다.

의미상으로
- 부모 타입일수록 더 추상적
- 자식 타입일수록 더 구체적

예를 들어
- Object -> 매우 추상적
- Animal -> 조금 구체적
- Dog -> 더 구체적

즉,
ObjectPrinter는 구체적인 대상이 아니라
더 넓은 범위의 공통 개념에 의존하는 구조이다.


[5] ObjectPrinter 구조의 핵심
----------------------------------------
ObjectPrinter는 다음 구조를 가진다.

- 매개변수로 Object를 받는다.
- 내부에서 toString()만 호출한다.
- 어떤 객체가 오든 공통 방식으로 출력할 수 있다.

예시)
print(Object obj)
-> obj.toString() 호출

이 구조가 좋은 이유
- 출력 대상이 무엇인지 미리 알 필요가 없다.
- Car가 오든 Dog가 오든 User가 오든 그대로 처리 가능하다.
- 새로운 클래스가 추가되어도 출력 코드는 그대로 유지된다.


[6] OCP 원칙과의 연결
----------------------------------------
OCP(Open-Closed Principle)
- Open: 확장에는 열려 있어야 한다.
- Closed: 변경에는 닫혀 있어야 한다.

Object + toString() 구조는 OCP를 잘 만족한다.

Open
- 새로운 클래스를 추가할 수 있다.
- 그 클래스에서 toString()만 오버라이딩하면 된다.

Closed
- 새로운 클래스가 추가되어도
  ObjectPrinter 같은 기존 출력 코드는 수정할 필요가 없다.

즉,
기능은 확장할 수 있지만
기존 클라이언트 코드는 바꾸지 않아도 된다.


[7] 왜 다형성을 잘 활용한 구조인가
----------------------------------------
이 구조는 다형성을 잘 활용한 대표 예시이다.

이유는 다음과 같다.

1. 다형적 참조 사용
   - Object 타입으로 모든 객체를 받을 수 있다.

2. 메서드 오버라이딩 사용
   - 각 객체가 toString()을 재정의해서
     자기 정보를 스스로 표현한다.

3. 클라이언트는 구체 타입을 몰라도 된다
   - 출력하는 쪽은 Object만 알면 된다.

즉,
"공통 부모로 받고"
"실제 동작은 각 자식이 오버라이딩으로 결정하는 구조"이다.


[8] System.out.println()도 같은 원리
----------------------------------------
지금까지 설명한 ObjectPrinter 구조는
사실 System.out.println()의 작동 방식과 매우 비슷하다.

println()도 객체를 받으면
내부적으로 toString()을 호출해서 출력한다.

즉,
System.out.println(객체);
이 코드는 결국 객체의 toString() 결과를 출력하는 방식이다.

그래서 자바에서는
객체를 그냥 println()에 넣기만 해도
알맞은 문자열 형태로 출력할 수 있다.


[9] 정적 의존관계와 동적 의존관계
----------------------------------------
1. 정적 의존관계
- 컴파일 시점에 결정되는 관계
- 코드상에서 어떤 타입을 사용하는지로 판단 가능
- 예: ObjectPrinter는 Object에 의존한다.

2. 동적 의존관계
- 런타임에 실제로 어떤 객체가 들어오는지에 대한 관계
- 실행해봐야 알 수 있다.
- 어떤 때는 Car 객체
- 어떤 때는 Dog 객체
- 어떤 때는 User 객체

정리하면
- 정적 의존관계 = 코드에 적힌 타입 기준
- 동적 의존관계 = 실행 시 실제 전달되는 객체 기준


[10] equals() - 동일성과 동등성
----------------------------------------
자바에서 "같다"는 개념은 2가지로 나뉜다.

1. 동일성(Identity)
- == 연산자로 비교
- 두 변수가 "완전히 같은 객체"를 가리키는지 확인
- 즉, 참조값이 같은지 비교

2. 동등성(Equality)
- equals() 메서드로 비교
- 두 객체가 "논리적으로 같은지" 확인

쉽게 말하면
- 동일성 = 물리적으로 같은 객체인가?
- 동등성 = 의미상 같은 객체인가?


[11] 동일성과 동등성 예시
----------------------------------------
예를 들어 회원 객체 2개가 있고
둘 다 회원번호가 "id-100" 이라고 하자.

- user1 = new User("id-100")
- user2 = new User("id-100")

이 경우
- user1 == user2 는 false
  -> 서로 다른 객체이기 때문

- 하지만 논리적으로는 같은 회원으로 볼 수 있다.
  -> 회원번호가 같기 때문

즉,
- 동일성은 다르다
- 동등성은 같을 수 있다


[12] Object의 기본 equals()
----------------------------------------
Object가 기본 제공하는 equals()는
사실상 == 비교와 같은 방식이다.

즉, 기본 equals()는
"논리적 비교"가 아니라
"참조값 비교"를 한다.

그래서 equals()를 재정의하지 않으면
결과는 동일성 비교와 거의 같다.

정리
- 기본 equals() = 동일성 비교
- 논리적 동등성 비교를 하려면 직접 오버라이딩해야 한다


[13] equals()를 재정의해야 하는 이유
----------------------------------------
동등성의 기준은 클래스마다 다르다.

예를 들어
- User는 id가 같으면 같은 사람으로 볼 수 있다.
- Product는 상품번호가 같으면 같은 상품으로 볼 수 있다.
- Member는 주민번호가 같으면 같은 회원으로 볼 수 있다.

즉,
논리적으로 무엇을 같다고 볼지는
각 클래스가 직접 정의해야 한다.

그래서 동등성 비교가 필요하면
equals()를 오버라이딩해야 한다.


[14] equals() 구현 시 핵심 흐름
----------------------------------------
equals()를 구현할 때 기본 흐름은 다음과 같다.

1. 자기 자신과 비교인지 확인
2. 비교 대상이 null인지 확인
3. 같은 클래스인지 확인
4. 실제 필드값 비교
5. 논리적으로 같으면 true, 아니면 false 반환

즉,
"객체 주소"가 아니라
"의미 있는 값"을 기준으로 비교해야 한다.


[15] equals() 구현 시 주의점
----------------------------------------
equals()는 단순히 형변환해서 필드만 비교하면
오류가 날 수 있다.

예를 들어
- null 비교 문제
- 타입이 다른 객체 비교 문제
- 잘못된 캐스팅 문제

그래서 실무에서는
IDE가 자동 생성해주는 equals()를 많이 사용한다.

핵심은
- 안전하게 타입을 확인하고
- 실제 비교 기준 필드를 사용해서
- 일관되게 비교해야 한다는 점이다.


[16] equals()가 지켜야 하는 규칙
----------------------------------------
equals()는 다음 성질을 만족해야 한다.

1. 반사성
- 자기 자신과 비교하면 true

2. 대칭성
- x.equals(y)가 true면
  y.equals(x)도 true

3. 추이성
- x와 y가 같고
  y와 z가 같으면
  x와 z도 같아야 한다

4. 일관성
- 객체 상태가 안 바뀌면
  여러 번 비교해도 결과가 같아야 한다

5. null 비교
- 어떤 객체든 null과 비교하면 false여야 한다

실무에서는 이 규칙을 외우기보다
IDE 생성 코드를 쓰고 의미를 이해하는 정도면 충분하다.


[17] equals()와 hashCode()
----------------------------------------
equals()는 보통 hashCode()와 함께 다룬다.

이유는 컬렉션(HashMap, HashSet 등)에서
두 메서드가 함께 동작하기 때문이다.

즉,
equals()만 재정의하고
hashCode()를 맞추지 않으면
컬렉션에서 예상과 다르게 동작할 수 있다.

지금 단계에서는
"equals()와 hashCode()는 보통 함께 간다"
정도만 기억하면 충분하다.


[18] 핵심 요약
----------------------------------------
1. Object가 없으면 객체별 전용 출력 메서드를 계속 만들어야 한다.

2. 그러면 출력 코드가 Car, Dog 같은 구체 클래스에 의존하게 된다.

3. Object를 사용하면 모든 객체를 공통 방식으로 받을 수 있다.

4. 각 객체는 toString() 오버라이딩으로 자기 정보를 표현한다.

5. 이 구조는 다형적 참조 + 메서드 오버라이딩을 잘 활용한 구조이다.

6. 따라서 새로운 클래스가 추가되어도 기존 출력 코드는 수정할 필요가 없다.

7. 이것이 OCP를 잘 지키는 구조이다.

8. System.out.println()도 같은 원리로 동작한다.

9. == 는 동일성 비교, equals()는 동등성 비교이다.

10. Object의 기본 equals()는 동일성 비교 방식이다.

11. 논리적 동등성 비교가 필요하면 equals()를 오버라이딩해야 한다.

12. equals()는 보통 hashCode()와 함께 고려해야 한다.


[19] 한 줄 정리
----------------------------------------
Object와 toString()은
모든 객체를 공통 방식으로 다루게 해주는 핵심 도구이고,
equals()는 객체가 "논리적으로 같은지" 비교하기 위해
필요할 때 재정의하는 메서드이다.
